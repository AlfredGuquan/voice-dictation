# PRD — Voice Dictation

## 背景与目标

用说话替代打字的 Mac 语音输入法。解决两个核心场景：长文写作时打字速度是瓶颈，
边看内容边记笔记时手眼被占用无法输入。现有云端听写工具（Typeless）不稳定且有持续
订阅成本，macOS 内置听写缺乏智能清洗。本项目自建转录管道，用自选云端 ASR + 轻量
LLM 清洗，在可靠性和成本之间取得平衡。

---

### F1: 核心听写管道

- 动机: 打字是长文写作和边看边记场景下的输入瓶颈，需要一条从语音到干净文字的完整管道
- 描述: 热键 toggle 开始/结束听写。录音期间持续捕获麦克风音频，结束后发送至云端 ASR
        获取原始转录，再经轻量 LLM 清洗（去填充词"嗯啊哦"、去重复词，保留原意不重组），
        最终一次性注入当前系统焦点输入框的光标位置。无焦点输入框时文字存入历史记录并
        复制到剪贴板，发系统通知告知用户。网络失败时通知用户，音频保留在历史中可重试
- 约束:
  - Mac only
  - 中文为主，夹杂英文词（中英混合识别）
  - 只做轻量清洗——去填充词、去重复，不改原意不重组句子
  - 云端 ASR + 云端 LLM，供应商可替换
  - 批量注入（结束后一次性），非流式
- 验收标准:
  - 按热键开始录音，再按结束，清洗后文字出现在当前焦点输入框光标处
  - 中文语句中夹杂英文词能正确识别
  - 填充词（嗯、啊、哦）和重复词被去除，原意保留
  - Esc 键取消录音，不产生任何文字输出
  - Cmd+Z 能撤销刚注入的文字
  - 焦点不在输入框时，文字复制到剪贴板并弹出系统通知
  - 网络失败时弹出通知，音频保留在历史中可重试
- 依赖: 无

### F2: 浮动药丸 UI

- 动机: 用户需要明确的视觉反馈知道系统在录音、在处理、还是已完成
- 描述: 录音开始时屏幕底部浮出药丸形控件。录音状态下中间显示音频波形动画，
        左侧 ✕ 按钮取消，右侧 ✓ 按钮确认结束。用户按第二次热键或点击 ✓ 后，
        波形变为从左到右的进度条表示处理中。处理完成后药丸自动消失
- 约束:
  - 药丸悬浮于所有窗口之上，不遮挡主要内容
  - ✕ 按钮和 Esc 热键功能一致（取消录音）
  - ✓ 按钮和第二次热键功能一致（结束并处理）
- 验收标准:
  - 按热键后药丸从底部浮出，显示波形动画
  - 说话时波形有动态响应
  - 点击 ✕ 或按 Esc 取消录音，药丸消失
  - 点击 ✓ 或再按热键，波形变为进度条
  - 处理完成后药丸自动消失
- 依赖: F1

### F3: 个人词库

- 动机: ASR 对专有名词和领域术语识别不准，需要用户可维护的词库提升准确率
- 描述: 支持两种词库条目：识别词（需正确识别的专有名词，如"顾权""Claude Code"）
        和替换映射（说 X 写成 Y，如"CC"→"Claude Code"）。词库以文本文件存储，
        同时提供应用内 UI 进行增删改管理。词库在 LLM 清洗阶段应用
- 约束:
  - 文件存储作为 source of truth，UI 读写同一文件
  - 词库变更即时生效，不需重启应用
- 验收标准:
  - 在 UI 中添加识别词后，听写时该词能被正确识别
  - 在 UI 中添加替换映射后，听写中说出触发词，输出为替换后的文字
  - 直接编辑词库文件后，应用能识别变更并生效
  - 在 UI 中能浏览、搜索、编辑、删除词库条目
- 依赖: F1

### F4: 主窗口与历史记录

- 动机: 需要查看和复用过去的听写内容，以及对比清洗效果来调优词库
- 描述: 菜单栏常驻应用图标，点击可打开主窗口。主窗口包含历史听写列表（按时间倒序），
        支持全文搜索。每条记录保留原始转录和清洗后文字两个版本。提供独立对比视图，
        左右并排展示原始/清洗版本。主窗口同时作为设置入口（热键配置、词库管理）
- 约束:
  - 菜单栏图标常驻，主窗口按需打开
  - 历史记录本地持久化存储
- 验收标准:
  - 点击菜单栏图标能打开主窗口
  - 每次听写完成后，历史列表自动出现新记录
  - 搜索框输入关键词能过滤匹配的历史记录
  - 点击某条记录能看到清洗后文字，可一键复制
  - 切换到对比视图，左右并排显示原始转录和清洗后文字
  - 设置页面能配置热键和管理词库
- 依赖: F1

### F5: 运行时通知改为 app 内 toast

- 动机: osascript display notification 触发系统通知中心太重，运行时错误提示在 pill 附近显示更轻量、上下文更连续；启动错误无 pill 可依，保留系统通知更稳
- 描述: 运行时 4 个通知场景（录音启动失败、pipeline 失败带/不带音频存档、无焦点剪贴板 fallback）改为 app 内轻量 toast。启动时 2 个场景（缺 API Key、辅助功能权限未授予）保留系统通知。Toast 样式和时长由 designer 定稿
- 约束:
  - 运行时 toast 需作为 floating panel 渲染，前台是其它 app 时仍可见
  - 不改启动期通知路径——app 刚拉起时 toast 渲染路径不稳
  - 视觉参数（位置、时长、配色）对齐 mockup 暖色体系
- 验收标准:
  - 运行时 4 个错误场景触发时不出现系统通知中心条目，而显示 app 内 toast
  - 启动期缺 API Key 或无辅助功能权限时仍出现系统通知
  - 无焦点复制到剪贴板场景下，即使前台是其它 app 也能看到 toast
- 依赖: F2

### F6: 药丸背景虚线边框修复

- 动机: 浮动 pill NSPanel 在部分系统/主题下渲染异常，出现方框虚线，视觉不干净
- 描述: 定位并修复 NSPanel 异常边框。根因可能是 borderless mask、背景层绘制顺序或 focusRing，由 tracer 实验确定
- 约束:
  - 修复不能破坏现有 pill 的 nonactivating + canBecomeKey=false + hidesOnDeactivate=false 配置
  - 多屏、浅色/深色模式、Retina/非 Retina 下都需验证
- 验收标准:
  - 浮动 pill 在浅色和深色模式下都不显示方框虚线
  - 多屏幕配置下切换显示器，pill 仍无边框异常
- 依赖: F2

### F7: Cmd+, 打开主窗口 Settings

- 动机: 菜单栏 Preferences 快捷键（Cmd+,）按下后错误地弹出文本输入框，应打开主窗口 Settings tab
- 描述: 修复菜单栏 Preferences 项的快捷键绑定和目标 action。根因（AppDelegate keyEquivalent 绑定错误 / 系统拦截 / responder chain 异常）由 tracer 定位
- 约束:
  - 主窗口已存在时复用，不重复打开
  - 打开后焦点定位到 Settings tab 而非历史 tab
- 验收标准:
  - 按 Cmd+, 打开主窗口并自动切到 Settings tab
  - 不弹出文本输入框或任何额外对话框
  - 主窗口已打开时 Cmd+, 将其 focus 并切到 Settings tab
- 依赖: F4

### F8: 历史对比视图 diff 标记

- 动机: mockup 设计了填充词红色删除线高亮，当前对比视图只做纯文本并排，用户看不出清洗删了什么
- 描述: 对比视图的原文展示时，用词粒度 diff 将被清洗掉的词标记为红色删除线。清洗后文本保持纯文本不加标记。中文按 tokenizer 切词，英文按空格切词
- 约束:
  - 词粒度（不是字符粒度）——整词整块标记
  - 只标删除，不标新增或改写
  - 颜色对齐 mockup.html 的 --diff-removed
  - 算法和分词库选型由 tracer 评估
- 验收标准:
  - 对比视图中原文里被清洗掉的词以红色删除线高亮显示
  - 清洗后文本展示为纯文本，不加任何颜色或删除线
  - 中英混合文本的分词结果与预期一致（英文按空格、中文按词边界）
- 依赖: F4

### F9: 听写快捷键可配置

- 动机: 右 Option 硬编码在 HotkeyManager，用户无法改为自己习惯的键位
- 描述: Settings 页新增"听写快捷键"录制控件，支持两种录音模式——单修饰键（右 Option / Fn / CapsLock 等）"按住说话"，组合键（Ctrl+Space / Cmd+Shift+D 等）"按一次开再按停"。检测到系统占用冲突时显示警告但允许保存。保存后不需重启 app 即可生效
- 约束:
  - 两种模式并存，由录入的键型自动选择
  - 冲突检测"只提示不拦截"——允许保存，不强制阻断
  - 热加载——修改后 HotkeyManager 立即切换到新键位
  - 录制 UI 交互细节由 designer 定稿
- 验收标准:
  - Settings 页有"听写快捷键"录制控件，点击后进入录制状态
  - 录入单修饰键（如右 Option）并保存后，按住该键开始录音、松开停止，整个流程立即生效
  - 录入组合键（如 Ctrl+Space）并保存后，按一次开始录音、再按一次停止并触发转录
  - 录入系统已占用的快捷键（如 Cmd+Space）时显示警告文本，但仍允许保存
  - 修改快捷键后不重启 app 即可用新键位触发听写
- 依赖: F4

### F10: API Key 热加载

- 动机: Settings 页保存 API Key 后必须重启 app 才生效，体验断裂
- 描述: OpenAIClient 改为从 Config 动态读取 key（或提供 refresh() 方法），Settings 保存后下一次转录请求立即用新 key
- 约束:
  - 不破坏现有调用方——client 接口保持一致
  - 保存瞬间不触发任何请求，只更新 key；下次转录时生效
- 验收标准:
  - Settings 页修改并保存 API Key 后不重启 app，下次按热键录音并转录注入能成功用新 key
  - 保存无效 key 后，下次转录触发鉴权错误并按现有错误路径反馈
- 依赖: F4

### F11: 进度条 Trickle 动画

- 动机: 当前 sweep 动画来回缩放，视觉不像"在进度"；用户期望处理期间有明确的"向右推进"感
- 描述: 处理态进度条改为 Trickle 风格——fill 从 0 快速填充到 ~70%，然后缓慢渐近到 ~95% 并停住（永不循环回 0），实际 ASR+清洗完成时跳到 100% 再隐藏 pill
- 约束:
  - 动画不循环、不重置——只向右前进
  - 渐近曲线在短处理（<1s）和长处理（>5s）下都要自然
  - 确切曲线参数（ease-out、时长）由 designer 定稿
- 验收标准:
  - 按热键录音并松开后，进度条 fill 从左快速向右推进
  - 处理期间 fill 只向右前进，不出现向左收缩或跳回 0
  - ASR 实际完成时 fill 达到 100% 再隐藏 pill
  - 短处理（<1s）和长处理（>5s）下动画都看起来自然
- 依赖: F2

### F12: 阶段耗时打点

- 动机: 当前全链路 7-8s 感知，但不知道 ASR/LLM/音频准备/注入 各段占比。没有分阶段数据就无法归因优化方向，也无法支撑未来换模型的对比
- 描述: DictationPipeline 每阶段打点记录耗时。HistoryStore.Record 新增 audioPrepMs / asrMs / llmMs / injectMs 字段。历史界面展开某条记录时展示四段耗时及总和
- 约束:
  - 不引入用户可感知的性能损耗（仅 Date() 差值，<1ms）
  - 旧历史条目（没有耗时字段）UI 显示为 "—"，不回填也不崩溃
  - 打点记录是副作用行为，不影响主流程的错误路径
- 验收标准:
  - 新产生的每条历史记录包含 audioPrepMs / asrMs / llmMs / injectMs 四段耗时
  - 展开历史列表某条记录能看到四段耗时（单位 ms，含总和）
  - 旧历史条目展开时耗时位置显示 "—"，不崩溃不空白
  - 连续听写 10 次后，历史界面可人工检视每次各段耗时数据
- 依赖: F4

### F13: Provider 抽象 + 模型升级

- 动机: 当前 7-8s 的主要贡献者是云端 whisper-1 + VPN 上传。本地 WhisperKit（Swift Package Manager 原生接入）消除 ASR 阶段网络依赖，目标全链路 <3s。同时为 F14 评估模式铺设 Provider 抽象层
- 描述: 抽象出 ASRProvider / LLMProvider 两个 Swift 协议。接入 5 个 Provider——WhisperKit（本地，新）、whisper-1（云端，既有重构）、GPT-4o-mini（既有重构）、GPT-4.1-mini（新）、Gemini 2.5 Flash（新）。默认生效组合为 WhisperKit + GPT-4.1-mini。HistoryStore.Record 新增 modelConfig 字段记录每条转录使用的 Provider 组合。设置页可切换生效 Provider，切换不需重启
- 约束:
  - WhisperKit 模型首次下载 ~1.5GB，需要进度 UI 且不阻塞现有 whisper-1 路径（下载期间继续用 whisper-1）
  - 缺少某 Provider 所需 API Key（如 GEMINI_API_KEY）时，该 Provider 在设置页灰掉且标注原因
  - Qwen3-ASR 仅作为 future candidate 记录在研究报告，本期不实装
  - 切换 Provider 失败时有明确错误提示，生效 Provider 回滚到上一个
- 验收标准:
  - 默认组合 WhisperKit + GPT-4.1-mini 下，全链路听写延迟在 VPN 环境下 <3s
  - 设置页能从 5 个 ASR Provider 选 1 个、从 3 个 LLM Provider 选 1 个
  - 切换 Provider 后下次听写立即生效，无需重启
  - WhisperKit 首次启动时自动下载模型，有可见进度条；下载期间听写仍能用 whisper-1 完成
  - 缺少 Gemini API Key 时该选项在设置页灰掉且显示原因说明
  - 新产生的历史记录保存本次使用的 modelConfig（ASR 名 + LLM 名）
- 依赖: F12

### F14: 评估模式

- 动机: 升级模型后要能持续回答"新模型真的更好了吗、好在哪层"。选型决策需要基于证据（原始音频、各阶段耗时、ASR 原文、LLM 清洗文的并排对比）而不是主观感受
- 描述: 主窗口侧边栏新增「评估」Tab。两部分组成——新建评估 + 历史评估运行列表。新建评估流程：从历史音频样本池多选 ≥1 条或现场录音 + 选两个 Provider 组合 → 并行调用 → N 样本 × 2 配置横评表格。每个 cell 展示 ASR 原文、LLM 清洗文、三段耗时、音频回放按钮。表格底部展示平均延迟汇总和「切换到配置 X」按钮。每次评估运行持久化到 EvaluationStore 供回顾
- 约束:
  - 音频样本池保留最近 50 条 FIFO 淘汰；用户可手动置顶某条，置顶不占额度
  - 评估模式下的现场录音不注入到光标（评估是旁路，不干扰正在使用的其他 app）
  - 单次评估固定 2 列对比；一侧失败另一侧成功时仍展示成功侧 + 失败侧标记错误原因
  - 单次 Provider 调用超时 30s（防 VPN 抽风）
  - 旧历史条目（无音频文件）不能作为评估样本
- 验收标准:
  - 主窗口侧边栏「评估」Tab 可见且可导航进入
  - 从历史样本池多选 ≥2 条、配置两个不同 Provider 组合能成功跑完评估
  - 现场录一段音频作为样本跑评估，录音过程不往其他 app 的焦点文本框注入文字
  - 对比表格每个 cell 展示 ASR 原文、LLM 清洗文、三段耗时和播放按钮
  - 汇总区展示两配置的平均延迟对比
  - 底部「切换到配置 X」按钮能把当前生效 Provider 组合切到该配置，下次日常听写立即用
  - 任一配置调用失败时另一侧不中断，失败侧显示错误原因而不是空白
  - 评估运行持久化，「历史评估运行」列表能查看过去跑过的 run 及其结果
  - 音频样本池超过 50 条时按 FIFO 淘汰最老的非置顶条目，置顶条目不受淘汰
- 依赖: F13

## Tracer Findings [2026-04-13]

全部 6 个集成点验证通过。

### 已验证的技术路径

| 集成点 | 方案 | 备注 |
|--------|------|------|
| 全局热键 | CGEvent tap + keyCode 61 (右 Option) | 需 Accessibility 权限，启动时检测 |
| 录音 | AVAudioEngine → WAV (48kHz Int16 PCM) | Whisper API 直接接受 |
| ASR | OpenAI Whisper API (`whisper-1`, language=zh) | 中文准确，英文大部分正确 |
| LLM 清洗 | GPT-4o-mini, temperature 0.3 | 去填充词+去重效果好，~¥0.0002/次 |
| 文字注入 | 剪贴板粘贴法 (保存剪贴板→写入→Cmd+V→恢复) | CGEvent 逐字符对中文有编码问题，不可用 |
| 浮动药丸 | NSPanel + nonactivatingPanel + canBecomeKey=false + hidesOnDeactivate=false | 不抢焦点，不出现在 Dock |

### 关键约束

- 文字注入必须用剪贴板法，需保存/恢复用户剪贴板内容，注入后延迟 ~200ms 再恢复
- ASR 对专有名词有系统性错误（如 Claude→Cloud），个人词库在清洗阶段注入修正
- NSPanel 必须设 hidesOnDeactivate=false，否则因 accessory app 不激活会立刻隐藏

### 验证方式

桌面原生应用，不适合 browser-qa。QA 采用 AppleScript 自动化（模拟热键 + 检查 TextEdit 内容）+ 人工验证药丸 UI。
